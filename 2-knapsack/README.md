# Отчёт по задаче Knapsack

## Как запустить:

Чтобы запустить основное решение, выполните команду `./run-all.sh` из директории `./2-knapsack`. Возможность запуска других решений не предусмотрена, но если очень надо, раскомментируйте нужную строку в `./src/solve.cpp`, поменяйте значение переменной `results` в `./run-all.sh` и перезапустите скрипт.

## Чекер:

Запускается скриптом `./run-all.sh`. Код: `./src/checker.py`. Вердикты чекера записаны в файлах `verdict.txt` в директориях `./results*`. Вердикт для основного алгоритма продублирован в корневую директорию задачи.

## Где искать отчёт:

- `./results`: результаты работы основного алгоритма.

- `./results-greedy`: результат работы жадного алгоритма.

- `./results-greedy-random`: результат работы жадного алгоритма с элементами рандомизации.

- `./results-beam`: результат работы beam search.

# Описание алгоритмов

Первым делом я написал жадный алгоритм (`./src/constructive-methods/greedy.hpp`). Он прошёл первый порог во всех тестах, кроме `ks_30_0` и `ks_10000_0`.

После этого я решил добавить в жадник рандома (`./src/greedy-random.hpp`). Для каждого предмета я посчитал отношение стоимости к весу, приплюсовал к нему случайное число от 0 до $t$ и отсортировал предметы по убыванию этого параметра, после чего набирал их жадно. Я запустил алгоритм несколько раз с разными $t$ от $1e-4$ до $4e4$, после чего выбрал лучшее найденное решение. Это решение прошло первый порог на всех тестах и достигло второго на `ks_200_0` и `ks_10000_0`.

После этого я запустил beam search (`./src/constructive-methods/beam_search.hpp`). Он пробил второй порог на первых трёх тестах.

Основным алгоритмом является динамическое программирование (`./src/dp.hpp`), работающее за $\mathcal O(nW)$ и находящее абсолютно точное решение. За счёт того, что у предложенных тестов $nW$ очень маленькое, это решение без проблем их все проходит.

В классической реализации динамического программирования, для восстановления ответа потребовалось бы хранить массив предков размером порядка $nW$ байт. Это слишком много &mdash; на большинстве компьютеров такой массив не влезет в оперативную память. Чтобы сэкономить, я храню массив неявно, аллоцируя только нужные мне ячейки (см. класс `Parent`) и храня их в виде дерева с корнем в `kInitial`. Благодаря этой оптимизации моё решение проходит все тесты, аллоцируя на массив предков максимум 250 Mb памяти.
